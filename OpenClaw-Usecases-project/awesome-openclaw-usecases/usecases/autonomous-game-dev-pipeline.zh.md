# 自主教育游戏开发流水线

## 痛点
**起源故事：** 一位"老派LAN玩家"爸爸想为他的女儿 Susana（3岁）和即将出生的 Julieta 创建一个安全、无广告、高质量的游戏门户网站。现有网站充斥着垃圾信息、攻击性广告和欺骗性按钮（黑暗模式），让他这个幼儿感到沮丧。

**挑战：** 构建"干净、快速、简单"的门户网站很简单，真正的挑战是在没有开发团队的情况下，为**40+个教育游戏**填充内容，这些游戏需要针对特定的发育阶段（0-15岁）。手动开发对于单独的父开发者来说太慢了，而在数十个游戏中保持一致性正在变成一场噩梦。

## 功能说明
此用例定义了一个"游戏开发者代理"，该代理自主管理游戏的创建和维护生命周期。该工作流程强制执行**"Bug优先"**政策，代理必须先检查并解决报告的Bug，然后才能实现新功能。

**效率：** 此流水线能够**每7分钟生产1个新游戏或Bug修复**。代理不知疲倦地处理41+个计划游戏的待办事项，在新游戏创建和纠正先前周期中检测到的问题之间交替进行。

当路径畅通时，代理：
1. **选择**：根据"轮询"策略从队列（`development-queue.md`）中识别下一个游戏，以平衡各年龄段的内容。
2. **实现**：为游戏编写HTML5/CSS3/JS代码，严格遵守`game-design-rules.md`（不使用框架，移动优先，离线功能）。
3. **注册**：自动将游戏元数据添加到中央注册表（`games-list.json`）。
4. **文档**：更新`CHANGELOG.md`和`master-game-plan.md`状态。
5. **部署**：处理Git工作流程：获取主分支，创建功能分支，使用约定式提交提交更改，然后合并回来。

## 提示词

此工作流程的核心是给予代理的**系统指令**。这个提示将LLM转变为一个遵守项目严格结构的自律开发者。

*（**注意：** 生产中实际使用的提示词是**西班牙语**（`es-419`），以符合项目的目标受众（拉丁美洲儿童）以及未来可能来自该地区的贡献者。以下版本为本文档翻译版。）*

```text
充当网页游戏开发和儿童用户体验专家。
您的目标是生产队列中的下一个游戏。

在开始之前，请阅读并分析以下上下文文件：

1.  Bug 上下文（最高优先级 - 关键）：
    @[bugs/]
    （检查此文件夹。如果有文件，您的任务是**只修复第一个文件**（按字母顺序）。忽略其余的Bug和游戏队列）。

2.  队列上下文（下一个游戏是哪个）：
    @[development-queue.md]
    （在"下一个游戏"部分中标记为[NEXT]的游戏。仅在无Bug的情况下）。

3.  设计规则（技术标准）：
    @[game-design-rules.md]
    （严格遵循这些规则：纯HTML/CSS/JS，文件夹结构，移动端响应式）

4.  游戏规格（机制和资源）：
    （根据游戏ID在games-backlog/中识别相应文件）

5.  中央注册表（集成）：
    @[public/js/games-list.json]
    （您必须在此文件中注册新游戏，以便它出现在首页）

任务：
0. **Bug优先！**：如果`bugs/`文件夹有内容，您唯一的优先级是修复**按字母顺序排列的第一个Bug**。创建一个`fix/...`分支，解决**该**Bug，更新状态，然后合并。**不要尝试同时修复多个Bug。**
   - 如果没有Bug，继续下一个游戏：

1. **同步**：`git fetch && git pull origin master`（关键）。
2. 创建新分支：`git checkout -b feature/[game-id]`。
3. 在'public/games/[game-id]/'中创建文件夹和文件。
4. 根据待办事项和设计规则实现逻辑和设计。
5. 在'games-list.json'中注册游戏（关键）。
6. 完成后：
    - 更新`CHANGELOG.md`并升级版本。
    - 更新`master-game-plan.md`和`development-queue.md`。
    - 记录更改：`git commit -m "feat: add [game-id]"`。
7. **交付**：
    - 推送：`git push origin feature/[game-id]`。
    - 请求合并到主分支。
    - 一旦合并到主分支，推送更改（`git push origin master`）。
```

## 所需技能
- **Git**：管理分支、提交和合并。
