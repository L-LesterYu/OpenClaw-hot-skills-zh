# 扩展模式

## 容量阈值

| 规模 | 条目数 | 策略 |
|-------|---------|----------|
| 小型 | <100 | 单一 memory.md，无命名空间 |
| 中型 | 100-500 | 拆分到 domains/，基本索引 |
| 大型 | 500-2000 | 完整命名空间层次结构，积极压缩 |
| 海量 | >2000 | 年度归档，HOT 层级仅保留摘要 |

## 何时拆分

在以下情况下创建新的命名空间文件：
- 单个文件超过 200 行
- 主题有 10+ 个不同的纠正
- 用户明确分离上下文（"对于工作..."，"在这个项目中..."）

## 压缩规则

### 合并相似纠正
```
BEFORE (3 entries):
- [02-01] Use tabs not spaces
- [02-03] Indent with tabs
- [02-05] Tab indentation please

AFTER (1 entry):
- Indentation: tabs (confirmed 3x, 02-01 to 02-05)
```

### 总结冗长模式
```
之前：
- 给 Marcus 发邮件时使用项目符号，保持在 5 条以内，
  不用行话，结论先行，他更喜欢早上发送

之后：
- Marcus 邮件：项目符号 ≤5，不用行话，结论先行，偏好上午
```

### 带上下文归档
当移动到 COLD 时：
```
## 已归档 2026-02

### 项目：old-app（自 2025-08 起不活跃）
- 使用 Vue 2 模式
- 更喜欢 Vuex 而非 Pinia
- CI 在 Jenkins 上（已弃用）

原因：项目已完成，模式不太可能再适用
```

## 索引维护

`index.md` 跟踪所有命名空间：
```markdown
# 记忆索引

## HOT（始终加载）
- memory.md: 87 行，更新于 2026-02-15

## WARM（匹配时加载）
- projects/current-app.md: 45 行
- projects/side-project.md: 23 行
- domains/code.md: 112 行
- domains/writing.md: 34 行

## COLD（归档）
- archive/2025.md: 234 行
- archive/2024.md: 189 行

上次压缩：2026-02-01
下次计划：2026-03-01
```

## 多项目模式

### 继承链
```
global (memory.md)
  └── domain (domains/code.md)
       └── project (projects/app.md)
```

### 覆盖语法
在项目文件中：
```markdown
## 覆盖
- 缩进：空格（覆盖全局的制表符）
- 原因：项目 eslint 配置要求使用空格
```

### 冲突检测
加载时，检查冲突：
1. 构建继承链
2. 检测矛盾
3. 最具体的优先
4. 记录冲突以供后续审查

## 用户类型适配

| 用户类型 | 记忆策略 |
|-----------|-----------------|
| 高级用户 | 积极学习，最少确认 |
| 普通用户 | 保守学习，频繁确认 |
| 团队共享 | 按用户命名空间，共享项目空间 |
| 注重隐私 | 仅本地，按类别明确同意 |

## 恢复模式

### 上下文丢失
如果智能体在会话中丢失上下文：
1. 重新读取 memory.md
2. 检查 index.md 以获取相关命名空间
3. 加载活动项目命名空间
4. 继续使用恢复的模式

### 损坏恢复
如果记忆文件损坏：
1. 检查 archive/ 以获取最近备份
2. 从 corrections.md 重建
3. 请求用户重新确认关键偏好
4. 记录事件以供调试
